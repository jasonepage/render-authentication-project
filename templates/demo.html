A live, fully-decoded walkthrough of a single WebAuthn round-trip. Every step shows the actual bytes flowing across the wire — challenges, CBOR-decoded attestation objects, COSE public keys, signatures, and the server-side verification checks.

Ephemeral. This page accepts any authenticator (security keys, Touch ID, Windows Hello, Android biometric) and writes nothing to the database. The credential lives in your session for the auth step, then disappears.

  1. 1

    Server generates a challenge

    Server · POST /demo/register/options

    pending

    The server creates 32 random bytes (the challenge) and a random user handle, then assembles a PublicKeyCredentialCreationOptions object. The challenge prevents replay attacks: the authenticator must sign over it, and the server checks that the same value comes back.

  2. 2

    Browser asks the authenticator to create a credential

    Browser → Authenticator · navigator.credentials.create()

    locked

    The browser passes the options to the platform's WebAuthn API. The authenticator (your security key, Touch ID, etc.) generates a fresh keypair, keeps the private key sealed inside, and returns the public key wrapped in an attestation object.

  3. 3

    Server decodes & verifies the attestation

    Server · POST /demo/register/verify

    locked

    The server CBOR-decodes the attestation object, parses the authData byte string (rpIdHash, flags, signCount, AAGUID, credentialId, COSE public key), converts the COSE key to a standard PEM, and runs the verification checks.

Run the registration walkthrough first — the authentication flow signs against the credential you registered above.

  1. 1

    Server generates a fresh challenge

    Server · POST /demo/auth/options

    pending

    A new 32-byte challenge for this assertion. The server also lists which credentials it expects (allowCredentials) — here, just the one you registered.

  2. 2

    Browser asks the authenticator to sign

    Browser → Authenticator · navigator.credentials.get()

    locked

    The authenticator looks up the matching private key, builds authenticatorData, concatenates it with SHA-256(clientDataJSON), and signs the result with the private key — which never leaves the device.

  3. 3

    Server verifies the signature

    Server · POST /demo/auth/verify

    locked

    The server checks that the challenge matches, reconstructs the signed payload (authenticatorData ‖ SHA-256(clientDataJSON)), and verifies the signature against the stored public key.
